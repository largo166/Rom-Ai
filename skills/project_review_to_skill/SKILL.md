# 项目复盘到 Skill 打包工作流

## 用途
把每日聊天、项目讨论、汇报推演、图像提示词、设计判断等零散内容，转化为可复用的工作流、知识条目、Skill 草案和 RMO-Ai 产品化线索。

## 触发场景
- 用户要求“复盘今天/最近/全部历史记录”
- 多次出现同一类项目判断、汇报、设计推导或图像提示词工作
- 用户明确说“这个要沉淀成 skill / 上传 GitHub / 封装进 RMO-Ai”

## 输入
- 聊天记录或会议纪要
- 项目资料、PDF、PPT、图片、方案截图
- Codex / GitHub / Obsidian / gbrain / 自动化运行结果
- 已有 Skills、Agents、Automations 清单

## 判断标准
仅当候选项满足以下条件时才进入打包：
1. 至少发生两次，或明显会高频复用且重复成本高；
2. 输入稳定、步骤可复用、输出明确；
3. 能提升速度、质量、一致性或可靠性；
4. 尚未被已有 Skill、Agent、Automation 充分覆盖；
5. 范围足够窄，不是泛泛的“大而全助手”。

## 标准步骤
1. 收集证据：按日期列出相关对话、文件、任务或自动化。
2. 识别重复动作：归纳为“触发场景—输入—动作—输出—停止条件”。
3. 选择最小形式：
   - Skill：稳定流程或操作手册；
   - Custom Subagent：专业角色或调查型任务；
   - Automation：定期检查、复盘、监控、提醒；
   - Skip：证据不足、一次性、敏感或过宽泛。
4. 先输出候选清单，不直接创建。
5. 只为高置信度候选生成草案。
6. 标记 GitHub 写入条件：仓库、分支、目录不明确时不得写入。
7. 形成 RMO-Ai 映射：模块、输入输出、价值、开发复杂度。

## 示例提示词
“回顾我最近 30 天的工作，找出重复性手动工作流。先输出候选清单，再只为高置信度内容生成 Skill 草案。不要重复创建已有自动化。”

## 边界条件
- 不得基于猜测创建资产。
- 不得把敏感项目原文直接写入公开仓库。
- 不得覆盖已有文件。
- GitHub 目标不明确时，只生成草案，不执行写入。
